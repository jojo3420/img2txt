# OCR/보정 품질 측정 하네스 및 두 실험 설계

작성일: 2026-07-13
성격: 구현 스펙. 측정 하네스(harness, 품질을 자동으로 재는 도구)와 그 위에서 돌릴 두 실험의 설계.
후속: 이 스펙 승인 후 writing-plans 스킬로 구현 플랜을 만든다.

## 1. 배경과 이번 세션의 재정의

원래 로드맵(`docs/superpowers/specs/2026-07-12-public-service-roadmap-design.md`)의 6.0 0단계 검증 게이트는 "사업성부터 검증"이 취지였다. 이번 세션은 사용자 지시로 방향을 바꿔, 사업성 판정 트랙(원가 합산, 광고 수익 추정, 통과 임계값, 대규모 대표 데이터셋)을 보류하고 순수 기술 품질 개선 두 가지에 집중한다.

- (1) OCR 한글 인식 성능과 정확도 측정지표 개선 (한글 우선)
- (2) LLM 보정 모델을 현행 gpt-5.5에서 가성비 좋은 모델로 바꾸고 성능 측정

두 작업의 공통 병목은 "정답 대비 품질을 숫자로 재는 도구"인데 현재 코드에 전무하다. 그래서 이 스펙의 1순위 산출물은 측정 하네스이고, OCR 개선과 LLM 비교는 그 위에 얹는 실험이다.

로드맵 결정과의 관계:
- 로드맵 결정 #5(0단계 게이트 최우선)는 폐기가 아니라 보류. OCR 품질/원가와 LLM 비용은 어차피 사업성 계산의 입력값이라, 기술을 먼저 확정하고 사업성 판정은 나중에 재개한다.
- 로드맵 결정 #6(저비용 후보 = mini)은 수정 필요. gpt-5.5-mini는 실재하지 않음이 확인됐다(아래 7.4).

## 4. 현재 코드 파악 (설계 근거)

파이프라인 단계: scanner → ocr → layout → assembler → corrector → writer.

- `img2txt/ocr.py:40` `recognize_page`가 Apple Vision(ocrmac 1.0.1)으로 이미지 1장을 OcrLine 목록(text, confidence, 좌표)으로 만들고 위→아래 정렬한다.
- `img2txt/ocr.py:50` 현재 호출은 `ocrmac.OCR(rgb_image, language_preference=[OCR_LANGUAGE]).recognize()`. `recognition_level`은 명시하지 않지만 ocrmac 기본값이 `accurate`(최고 정확도)라 이미 최선을 쓰는 중.
- 정답과 비교해 정확도를 재는 코드는 파이프라인 어디에도 없다. `confidence` 값만 존재.
- `img2txt/backends/cli.py:58`에 codex 호출 모델명 `gpt-5.5`가 하드코딩돼 있고, `correct_batch`의 `model` 파라미터는 "미사용"으로 무시된다(cli.py:90 주석). `run_correct`가 넘기는 `model`(cli.py:140)은 로그/기록용일 뿐 실호출과 무관.

### ocrmac 옵션 (조사 결과, 소스코드 확인)

| 파라미터 | 기본값 | 의미 |
|----------|--------|------|
| framework | vision | 인식 엔진(vision 또는 livetext) |
| recognition_level | accurate | accurate(정확-느림) 또는 fast. vision 전용 |
| language_preference | None(코드에선 ko-KR) | 언어 힌트 |
| confidence_threshold | 0.0 | 최소 신뢰도 필터. vision 전용 |

함의: 현행이 이미 accurate라 인식 옵션 튜닝의 개선 여지는 작다. `confidence_threshold`를 올리면 낮은 신뢰도 글자를 버려 글자 누락(정답 글자 손실) 위험이 있어 개선이 아니라 트레이드오프다. 따라서 (1)의 지렛대는 옵션이 아니라 입력 전처리에 있다.

## 5. 측정 하네스 설계

### 5.1 컴포넌트

`scripts/bench_ocr.py`를 진입점으로 두고 다음 책임을 분리한다(단일 책임). 재사용 로직은 `img2txt/` 하위 모듈로 두어 단위 테스트 가능하게 한다.

- 데이터 로더: 데이터셋 이미지와 정답 라벨을 읽어 "페이지 이미지 + 정답 페이지 텍스트" 쌍으로 만든다. AI Hub 라벨이 문단 단위인지 글자 bbox 단위인지에 따라 정답 텍스트 재구성 방식이 달라지므로, 라벨 어댑터를 별도 함수로 둔다.
- 파이프라인 러너: 한 이미지에 대해 3지점 출력을 만든다. 원시 OCR(ocr.py의 줄 텍스트 이어붙임), 조립본(assembler 출력), 보정본(corrector 출력). 각 출력에 page_id를 부여하고, 보정본은 segment_id(문단 순번)와 문단 순서를 함께 기록해 채점 시 정답과 재결합/정렬을 고정한다. 원시 OCR 이어붙임 순서는 파이프라인의 좌표 정렬(sort_lines_top_to_bottom) 결과 순서를 읽기 순서로 삼는다.
- 정규화기: 채점 전 텍스트 전처리. NFC 유니코드 정규화, 연속 공백/줄바꿈 정리, 문장부호 종류 통일.
- 채점기: 정규화된 출력과 정답으로 CER(글자 오류율)와 WER(단어 오류율)을 편집거리(Levenshtein)로 계산.
- 리포터: 지점별 요약 통계와 페이지별 상세를 리포트 파일로 남긴다.

### 5.2 데이터 흐름

이미지 + 정답 라벨 → (파이프라인 러너) 3지점 출력 → (정규화기) 출력-정답 각각 정규화 → (채점기) 지점별 CER/WER → (리포터) 요약 + 페이지별 상세.

### 5.3 채점 규칙 (D5)

- 정규화: NFC 적용(한글 자모 조합 형태를 통일하지 않으면 같은 글자가 오류로 잡힘, 필수). 연속 공백은 하나로, 줄바꿈은 공백으로.
- 이중 점수: 엄격 CER(원문에 가까운, 공백/부호 최소 정규화)와 관대 CER(공백 정리 + 부호 매핑까지 적용)를 함께 계산해 병기한다. 정규화가 실제 오류를 숨기는지 엄격 점수로 확인한다.
- 부호 매핑 표: 관대 점수 계산 시 아래 표로 종류만 통일하되 삭제하지 않아 부호 오류도 잡는다. (구현 시 최종 확정, 초안)

| 원본 예 | 통일 후 |
|---------|---------|
| 유니코드 큰따옴표(좌우 곡선형) | ASCII 큰따옴표 |
| 유니코드 작은따옴표(좌우 곡선형) | ASCII 작은따옴표 |
| 각종 대시(en/em/붙임표) | ASCII 하이픈 |
| 말줄임표 1글자 | 마침표 3개 |
| 전각 숫자/영문 | 반각 |

- 지표: 주지표 CER, 보조 WER. 한글은 띄어쓰기가 불안정해 WER가 과대반영되므로 CER를 우선 본다.
- 집계 공식: 최종 점수는 micro 평균(모든 페이지의 편집거리 합을 정답 글자수 합으로 나눔)으로 확정한다. 페이지별 CER과 macro 평균/중앙값은 분포 참고용으로 리포트에 함께 남기되 판정 기준은 micro다.
- 읽기 순서: 채점은 순서를 반영하는 편집거리라, 원시 OCR 지점은 위 5.1의 좌표 정렬 순서를, 정답은 데이터셋 라벨의 자연 읽기 순서를 기준으로 정렬해 비교한다.

### 5.4 오류 처리

- OCR 빈 결과 페이지: 정답이 있는데 출력이 비면 정답 글자 전체를 누락 오류로 계산한다(제외 금지). 제외하면 성능이 실제보다 좋아 보이기 때문. 빈 결과 플래그를 리포트에 남긴다.
- 정답과 출력 매칭: 페이지 파일명/식별자 기준으로 1:1 정렬. 매칭 실패는 데이터 오류이므로 기본 동작은 실험 중단이고, 명시적 옵션(--allow-skip 등)이 있을 때만 스킵하며 카운트를 남긴다.
- 라벨 형식 미확인 상태: 다운로드 전이라 라벨 어댑터는 실제 구조 확인 후 확정한다(8절 후속 항목).

### 5.5 규모와 테스트

- 규모: 소규모 스모크 30~50페이지. 하네스가 3지점을 정답과 올바르게 정렬해 채점하는지부터 검증한다. 층화 표본이나 규모 확대는 다음 실행 단계 결정.
- 테스트: 정규화 함수(NFC/공백/부호)와 채점 함수(CER/WER)를 알려진 입출력으로 단위 테스트한다. 소량 골든 페이지로 하네스 전체 스모크. 단순 CLI 연결부는 과도하게 테스트하지 않는다.

### 5.6 리포트 스키마

리포트는 재현 가능하도록 페이지별 레코드를 구조화 형식(JSON Lines 또는 CSV)으로 남긴다. 최소 필드:

- 실행 메타: run_id, 데이터셋 식별자와 해시, macOS/ocrmac/codex CLI 버전, 프롬프트 버전, 모델 식별자, 전처리 설정, 실행 시각.
- 페이지 레코드: page_id, 측정 지점(raw/assembled/corrected), 정답 텍스트, 원문 출력, 정규화 출력, CER(엄격/관대), WER, 처리 시간(ms), 빈 결과 플래그, 오류 상태.
- 요약: 지점별 micro CER/WER, 악화 페이지 비율 등 부작용 지표(7.2), 실패율.

## 6. (1) OCR 전처리 실험 설계

목표: 현행 Apple Vision을 유지한 채 입력 이미지를 개선해 원시 OCR 정확도를 올린다.

- baseline: 현행 파이프라인(전처리 없음)의 원시 OCR 지점 CER를 기준선으로 측정.
- 실험 레버(각각 baseline 대비 하네스로 A/B 측정): 대비 향상, 해상도 업스케일, 기울기(deskew) 보정. 각 레버는 독립 적용해 효과를 분리 측정한다.
- `confidence_threshold` 필터는 개선이 아닌 트레이드오프로 분류. 순효과(누락 증가 대 오인식 감소)를 측정으로 확인만 한다.
- 효과 수치는 전부 하네스 실측으로 확정한다. 조사 과정에서 나온 사전 추정치("+5~10%" 등)는 근거가 없어 스펙과 판정에서 배제한다.
- 사전 교정은 이번 범위 밖. 실측에서 드러난 한글 오류 패턴을 보고 다음 세션에 설계한다.

## 7. (2) LLM 보정 가성비 비교 실험 설계

목표: 보정 모델을 가성비(가격+속도) 좋은 모델로 바꾸되 품질 하한을 지킨다.

### 7.1 리팩터 (선행 작업)

`img2txt/backends/cli.py:58`의 모델명 하드코딩(`-m gpt-5.5`)을 제거하고, 모델을 주입 가능하게 파라미터화한다. `correct_batch`의 `model` 파라미터가 실제 CLI 호출에 반영되도록 연결한다. 이 리팩터가 없으면 5.5와 저비용 모델을 같은 하네스로 비교할 수 없다.

### 7.2 비교 구성

- 품질 기준선(앵커, 채택 후보 아님): gpt-5.5.
- 가용성 확인 후 후보: gpt-5.6-luna(최신 세대 경량), gpt-5.4-mini(극저비용). 실호출 확인 전까지 확정 후보로 취급하지 않는다(7.4).
- 측정: 같은 문단 집합에 대해 조립본 대비 보정본의 CER 개선폭 + 호출 시간(속도) + 호출 비용을 지점 측정.
- 부작용 지표(평균 CER만으로는 과잉 보정을 놓치므로 필수): 악화 페이지 비율(보정 후 CER이 오른 페이지), 최악 악화폭, 삭제/추가 글자량, 실행 실패율, 기존 corrector 길이 가드 차단율(GUARD_BLOCKED 비율). 평균 개선이 있어도 악화 지표가 크면 채택에서 감점한다.
- 재현 고정: 프롬프트, 모델 식별자, codex CLI 버전, 재시도 규칙을 고정한다. LLM은 비결정적이라 codex의 온도/시드 제어 가능 여부를 확인하고, 불가하면 동일 입력 반복 실행으로 변동폭을 리포트한다.
- 비용 측정: codex exec는 실행당 토큰 총량을 출력한다(실측 확인). 입출력 토큰 분리와 단가 확인은 8절 선행 확인 항목에서 처리한다.

### 7.3 채택 판정 규칙 (D8)

- 우선순위: (1) 품질 하한 통과 -> (2) 페이지당 비용 최저 -> (3) 속도로 동률 판정. 품질 하한을 넘긴 모델들 중 페이지당 비용이 가장 낮은 것을 택하고, 비용이 같으면 빠른 쪽을 택한다.
- 품질 하한: 조립본 대비 보정본 CER 개선폭이 기준선(gpt-5.5) 대비 허용 열화 이내이고, 부작용 지표(7.2)가 상한 이내일 것.
- 허용 열화 탐색-평가 분리: 결과를 보고 합격선을 바꾸는 것을 막기 위해, 소규모 스모크(탐색셋)로 허용 열화와 부작용 상한을 먼저 확정한 뒤, 보지 않은 별도 평가셋(홀드아웃)에 그 기준을 적용해 최종 판정한다. 사후 조정 금지. 임계값은 조정 가능한 파라미터로 두되 실험 전에 고정한다.

### 7.4 모델 식별자 검증

gpt-5.5-mini는 실재하지 않는다(GPT-5.6 세대에서 mini 접미사 폐기, Sol/Terra/Luna 계층으로 전환). codex CLI(codex-cli 0.144.1)가 `-m <MODEL>` 옵션을 받는 것은 `codex exec --help` 실측으로 확인됐으나, 개별 모델명(luna, 5.4-mini)은 웹 문서 기반이다. 구현 착수 시 `codex exec -m gpt-5.6-luna` 등 실호출로 최종 검증한다.
