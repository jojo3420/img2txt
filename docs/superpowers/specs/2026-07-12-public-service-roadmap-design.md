# 공개 수익형 서비스 전환 로드맵 (Roadmap Spec)

작성일: 2026-07-12
최종 개정: 2026-07-12 (자체 적대적 리뷰 + Codex 교차 리뷰 반영)
성격: 개별 구현 스펙이 아니라 큰 그림을 고정하는 분해(decomposition) 문서. 각 서브프로젝트는 별도 스펙 -> 플랜 -> 구현 사이클을 따르며, 각자 산출물/핵심지표/선행의존/통과조건/실패대안을 자기 스펙에서 정의한다.

## 1. 목표

개인용 OCR 도구를 외부 사용자에게 공개하는 수익형 웹서비스로 전환한다. 수익원은 광고와 구독이다.
- 무료 사용자: OCR 변환 사용 + 광고 노출.
- 구독 사용자: OCR + LLM 보정 기능 사용(로그인 필수).

## 2. 토대 블로커 (리스트에 없었지만 거의 모든 것을 가로막는 전제)

공개 이전에 반드시 풀어야 하는 두 가지. 근거는 현재 코드다.

- OCR 엔진이 macOS 전용이다. `img2txt/ocr.py`는 `ocrmac`(Apple Vision)에 의존한다. 리눅스 서버에서 동작하지 않으므로, "macOS로 호스팅해 Vision 유지" 대 "크로스플랫폼/클라우드 엔진으로 교체" 결정이 선행되어야 한다. 이 결정이 OCR 품질(항목 5)과 다국어 OCR(항목 6)을 동시에 좌우한다.
- 서버측 LLM 보정 API 백엔드가 스텁이다. `img2txt/backends/api.py`(`ApiBackend`)는 원문을 그대로 반환한다. 현재 실제로 도는 보정은 `img2txt/backends/cli.py`가 로컬 `claude`/`codex` CLI를 운영자 개인 구독 계정으로 subprocess 실행하는 방식이라, 다중 사용자 공개 서비스에 사용할 수 없다. 운영자 API 키로 호출하는 서버측 백엔드를 실제 구현해야 한다.

추가 제약: 잡 상태가 인메모리(`server/jobs.py`의 `JobStore.jobs`)이고 산출물은 로컬 `./jobs`에 저장되며 보관/정리 로직이 없다. 동시성은 `MAX_CONCURRENT_JOBS=2`, `ThreadPoolExecutor` 기반이라 다중 인스턴스/공개 트래픽/재시작에 취약하다.

## 3. 수익 모델 (초안)

- 무료 티어: OCR 변환, 광고 노출, LLM 보정 불가.
- 구독 티어: OCR + LLM 보정, 광고 제거(정책 결정 필요).
- 결제 수단, 가격, 무료 한도(월 페이지 수 등)는 서브프로젝트 E에서 확정.

경고(단위 경제성): "페이지당 광고 수익이 페이지당 원가(클라우드 OCR + LLM API + 스토리지/트래픽)를 넘는다"가 성립해야 흑자다. 성립하지 않으면 무료 사용자가 늘수록 적자다. 이 검증이 다른 어떤 기능보다 먼저다(0단계, 섹션 5.1과 6.0).

## 4. 서브프로젝트 분해

각 서브프로젝트는 자기 스펙에서 통과 조건(수치 포함)과 실패 시 대안을 정의한다.

### 토대 (공개의 전제)

- A. OCR 엔진/배포 결정 (항목 5, 6 백엔드 포함)
  - 목적: 서버에서 돌아가는 OCR 파이프라인 확정 + 한국어/영어/일본어 지원 + 품질 개선.
  - 후보: macOS 호스팅(Vision 유지), Tesseract, PaddleOCR, 클라우드 OCR(Google Vision/Azure/Naver Clova).
  - 평가 기준(동일 표본으로 실측): 언어별(한/영/일) 정확도, 표/세로쓰기/저화질/회전 이미지 처리, 페이지당 처리시간, 페이지당 비용. 세로쓰기(일본어)는 인식뿐 아니라 레이아웃 재구성(`assembler`/`layout`)까지 영향을 주는 깊은 난제라 별도로 확인한다.
  - 품질 회귀 리스크: Apple Vision은 한국어/일본어 인식이 강하다. 오픈소스 엔진으로 옮기면 품질이 떨어질 수 있으니 "개선"을 전제하지 말고 Vision 대비 회귀를 먼저 확인한다. Vision급이 필요하면 클라우드 OCR이 유력하나 원가가 오른다(3, 5.1 연동).
  - 모바일/HEIC 후속: 아이폰 기본 HEIC(`pillow-heif` 필요) 지원을 공개 전 후속 과제로 포함.

- B. 서버측 LLM 보정 API 구현 + 모델 선택 (항목 4)
  - 목적: `ApiBackend` 실제 구현(운영자 API 키). 고품질 모델 대 저비용 모델을 품질/비용/속도로 벤치마크 후 결정(현재 후보: 사용자 지정 GPT-5.5 대 mini). 특정 모델명은 낡기 쉬우니 실제 가용 모델/가격은 이 단계에서 확인한다.
  - 판정 기준: 보정 품질 저하가 크지 않으면 저비용/고속 모델 채택.
  - 다국어 영향: 현재 보정 프롬프트는 한국어 전용이다. 영어/일본어 보정을 지원하려면 언어별 프롬프트/처리가 필요하므로 항목 6은 OCR만이 아니라 여기에도 걸린다.
  - 열린 결정: 벤더, 배치 크기, 타임아웃, 레이트리밋, 실패 폴백.

- C. 배포/호스팅 + 운영(observability)
  - 목적: 인메모리/로컬 구조를 서버 배포 가능한 형태로. 오브젝트 스토리지, 잡 큐/상태 영속화(Celery/RQ + 워커), 도메인/HTTPS, 동시성/스케일.
  - 운영: 오류 추적, 비용 알림, 잡 지연 감시, 백업, 재시도/중복 실행 방지(멱등성), 장애 시 사용자 안내를 포함한다.
  - 의존: A(호스팅 OS 결정)에 영향받음.

### 제품/수익

- D. 로그인/계정 (항목 1, 2)
  - build-vs-buy: 자체 인증 구현은 보안 리스크가 크다. 관리형 인증(Clerk/Auth0/Firebase Auth)을 우선 검토.
  - 비로그인 이력 병합 보안: 브라우저 localStorage의 job_id를 로그인 계정에 합칠 때, 브라우저 식별자만으로 병합하면 남의 결과를 가로챌 수 있다. 소유권을 확인하는 일회용 토큰(업로드 시 발급) 같은 장치가 필요하다.
  - 통합: 디스크 복구(job.json) 방식은 C의 스토리지 결정(로컬 대 오브젝트 스토리지, 다중 인스턴스)에 종속되므로 C 확정 후 마무리.

- E. 구독/결제 + 권한 게이팅 (항목 1)
  - 목적: 구독 상태 관리, LLM 보정을 구독자에게만 허용하는 서버측 권한 검증. 결제는 관리형(Stripe) 우선.
  - 원칙: 프런트 숨김과 별개로 백엔드 권한 검증 필수.
  - 사용량 상한: 사용자별 월 페이지/토큰 쿼터와 스펜드 캡으로 원가 폭주 차단.
  - 결제 예외: 웹훅 중복 처리(멱등), 결제 실패, 해지, 환불, 요금 변경, 권한 만료, 세금/영수증 처리를 명시.

- F. 광고 연동
  - 목적: 무료 티어 광고. 광고 네트워크 선택, 배치, 구독자 광고 제거 정책, 광고/쿠키 동의(H와 연동).

- G. 문구 중립화 + UI 다국어 i18n (항목 3, 6의 UI)
  - G1(공개 전 필수, 저비용): "한글 책" 등 특정 문구를 중립적으로 수정. 독립적이라 언제든 먼저 처리.
  - G2(확장 단계): UI를 한국어/영어/일본어로 제공하는 i18n 프레임워크 도입.

### 공개 전 안전장치 (신규, Codex 제안 반영)

- H. 컴플라이언스 및 운영 안전장치 (공개 전 필수 트랙)
  - 법률/개인정보: 이용약관, 개인정보처리방침, 저작권 침해 신고/삭제(테이크다운), 쿠키/광고 동의, 삭제 요청 처리, 제3자(OpenAI/클라우드 OCR) 전송 고지.
  - 데이터 수명주기: 원본 이미지/OCR 결과/LLM 요청-응답/로그 각각의 보관 기간과 자동 삭제 정책(원본은 짧게 보관 후 자동 삭제 원칙).
  - 업로드 강건화: 매직넘버 검증(이미 구현)에 더해 압축 폭탄(decompression bomb) 방어, EXIF 위치정보 제거, 해상도/페이지 수 상한, 손상 파일 처리.
  - 관리자 최소 기능: 사용자 차단, 쿼터 조정, 결제 상태 확인, 실패 잡 재처리, 삭제 요청 대응.

## 5. 리스크와 컴플라이언스

### 5.1 단위 경제성 (사업 성립의 핵심, 0단계 게이트)
페이지당 원가(클라우드 OCR + LLM API + 스토리지/트래픽)와 페이지당 광고 수익을 실측/추정한다. 광고 수익 환산은 RPM(1000 노출당 수익)에 광고 노출률, 광고 차단율, 국가별 단가를 반영해 페이지당 수익으로 낮춰 잡는다. 수요 검증은 광고 수익만으로 부족하니 방문자 수, 재방문율, 완료율, 페이지당 비용, 구독 의향을 함께 본다. 통과/중단 기준을 수치로 못박고, 적자 구조면 모델을 재설계한다(무료 한도 축소, 워터마크, 저가 엔진 등). 큰 구현 착수 전 스파이크로 끝낸다.

### 5.2 법률/개인정보
저작권 있는 책 스캔의 침해 조력 위험(약관/테이크다운으로 대응), PIPA/GDPR 준수와 보관-파기 정책, 제3자 데이터 처리 고지. 구체 실행은 서브프로젝트 H.

### 5.3 남용/보안
인증 없는 공개 업로드는 불법 콘텐츠, 초대용량/악성/압축폭탄 파일, 과금 폭탄의 벡터다. 사용자/IP별 레이트리밋과 쿼터, 업로드 강건화(H), 오류 메시지의 민감정보 비노출을 갖춘다. 보안 체크리스트는 `rules/security.md`를 따른다.

## 6. 의존 관계와 권장 순서

### 6.0 0단계 검증 게이트 (모든 경로의 최우선)
대표 이미지 데이터셋으로 OCR 품질/속도/비용을 측정하고 광고 수익 가정을 보수적으로 계산한다. 통과/중단 기준을 수치로 정한다(5.1). 통과해야만 본 구현에 들어간다.

### 6.1 완전 구축 순서 (기능 완결 우선)
0단계 게이트 -> A, B (토대) -> C (배포/운영) -> D (로그인) -> E (결제/게이팅) -> F (광고) -> G/H.

### 6.2 MVP-우선 대안 (권장, 리스크 최소화)
0단계 검증 게이트 -> A/C 최소 구축 -> G1 문구 중립화 + H 공개 안전장치 -> 제한 공개 -> F 광고 실험 -> B/D/E 구독 티어 -> G2/항목6 다국어 확장.
장점: 가장 비싼 부분(결제/인증)을 수요가 증명된 뒤에 짓고, 공개 전 필수 안전장치(H)를 순서에 포함한다. 단점: LLM 보정이라는 차별화 기능이 초기엔 빠진다.

권장: 6.2. 단, 어느 경로든 0단계 게이트가 최우선이다.

예외(틈새 작업): G1(문구)과 항목 4(모델 스왑 벤치마크)는 저비용/독립적이라 순서와 무관하게 진행 가능.

## 7. 상태

- 완료: 입력 포맷 확장(JPEG에 더해 PNG/WEBP/TIFF 허용, HEIC 제외). 백엔드 테스트 129개 통과, 프런트 빌드 성공.
- 파킹: "비로그인 내 결과"(localStorage 이력 + 디스크 복구) 설계는 C(스토리지)와 D(로그인, 소유권 병합)에 종속되어 그때 통합 재개.

## 8. 확정이 필요한 전략 결정 (다음 단계 입력값)

1. 단위 경제성: 페이지당 원가와 광고 RPM 환산, 무료 티어 흑자 여부, 통과/중단 수치. -> 6.0 스파이크.
2. 구축 경로: 6.1 완전 구축 대 6.2 MVP-우선.
3. 호스팅 OS: macOS(Vision 유지) 대 리눅스(엔진 교체). -> A.
4. OCR 엔진과 언어별 품질/비용(Vision 대비 회귀, 세로쓰기, HEIC). -> A.
5. LLM 벤더/모델(고품질 대 저비용)과 비용 상한, 다국어 보정. -> B.
6. 인증/결제 build-vs-buy(관리형 대 자체). -> D, E.
7. 구독 가격/무료 한도/사용량 쿼터. -> E.
8. 법률/개인정보 정책, 데이터 보관-삭제 기간. -> H.
9. 구독자 광고 제거 여부. -> F.

## 9. 다음 단계

이 로드맵 승인 후, 최우선으로 6.0 0단계 검증 게이트(단위 경제성 + OCR 품질/비용 스파이크)를 수행한다. 통과하면 서브프로젝트 A(OCR 엔진/배포 결정)부터 개별 브레인스토밍 -> 스펙 -> 플랜으로 진행한다.
