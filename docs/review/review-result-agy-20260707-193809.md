### 구체적 피드백

1. **[Blocker] `LENGTH_GUARD_RATIO` (10%)의 짧은 문단 오작동 위험**
   - 문단의 길이가 매우 짧은 경우(예: 20~30자), 10% 가드는 2~3자에 불과합니다. 설계상 핵심 타깃인 "20 세기" (5자) $\rightarrow$ "20세기" (4자) 처럼 공백 하나만 제거해도 길이 변화가 20%가 되어 가드에 걸려 보정이 차단됩니다.
   - **대책**: 최소 허용 절대 글자 수 편차(예: `min_diff_chars = 5` 또는 `max(5, length * 0.1)`)를 도입하여 짧은 문단이 가드에 막히지 않도록 공식을 보완해야 합니다.

2. **[Blocker] macOS 전용 라이브러리(`ocrmac`) 의존성에 따른 플랫폼 이식성 결여**
   - `ocrmac`은 macOS의 Vision 프레임워크에 강하게 결합되어 있어 Windows, Linux 또는 일반 CI(Github Actions 등) 환경에서 패키지를 import하는 것만으로도 에러가 발생합니다.
   - **대책**: `ocr.py` 내부에서 `ocrmac`을 동적 임포트(lazy import)하고 `ImportError` 발생 시 대체 목(Mock) 객체나 에러 메시지를 반환하게 하여, OCR 기능 외의 `correct` 도구 등은 타 OS에서도 개별적으로 작동할 수 있게 설계해야 합니다.

3. **[Edge Case] 다단(Multi-column) 레이아웃 및 삽화/도표 페이지 처리 누락**
   - 본문이 2단으로 구성되어 있거나 큰 삽화 및 표가 들어간 페이지의 경우, 단순 y좌표 정렬만 적용하면 좌우 칼럼의 텍스트 줄이 교차하여 심각하게 섞이는 문제가 발생합니다.
   - **대책**: 본 도구가 단일 단(Single-column) 텍스트 전용임을 한계점(Limitations)에 명시하고, x좌표의 다중 군집(Multi-cluster) 분포 분석 등을 통해 다단이 감지될 경우 warning을 남기는 안전장치가 필요합니다.

4. **[Exception] Ollama 로컬 실행 시의 지연 시간(Latency) 및 병렬화 부재**
   - 책 한 권 분량(일반적으로 수백 개 문단)을 단일 스레드로 동기 순차 요청할 경우, 로컬 GPU 환경에 따라 수십 분 이상의 실행 시간이 소요되어 CLI 사용자 경험을 크게 저해할 수 있습니다.
   - **대책**: Ollama가 다중 요청(`OLLAMA_NUM_PARALLEL`)을 지원하므로, `asyncio`나 스레드 풀을 이용하여 문단 보정 작업을 제한된 동시성(예: `max_workers=3`) 하에 병렬 처리할 수 있도록 설계를 확장해야 합니다.

5. **[Edge Case] 파일 시스템 인코딩 명시 누락**
   - `pages/*.txt`, `book.txt`, `corrections.log` 등을 작성하고 읽을 때 인코딩을 명시하지 않으면, 한글 Windows 환경(cp949 기본값) 등에서 깨짐 현상이 발생할 수 있습니다.
   - **대책**: 모든 파일 쓰기/읽기 작업(`open()`) 시 `encoding='utf-8'` 옵션을 필수로 적용하는 규칙을 명문화해야 합니다.

### 개선 제안

1. **(우선순위: 상) LLM 보정 프롬프트에 퓨샷(Few-shot) 예시 포함**
   - 단순히 지시문(System Prompt)으로 제약을 주는 것보다, 실측 데이터에서 발견한 주요 오류 패턴("경단로" $\rightarrow$ "결단코", "20 세기" $\rightarrow$ "20세기")을 2~3쌍의 Few-shot 예시로 프롬프트에 제공하면 보정 정확도와 일관성이 대폭 향상됩니다.

2. **(우선순위: 중) 임계값 캘리브레이션을 위한 드라이런(Dry-run) 모드 제공**
   - `FOOTER_BAND`, `INDENT_MIN` 등의 파라미터는 책의 판형마다 조정이 필요할 수 있습니다. 특정 1~2페이지를 대상으로 감지된 좌표와 꼬리말 제거 영역을 콘솔에 덤프해 주는 시각적 디버깅 옵션(예: `convert --dry-run`)을 CLI 사양에 추가하는 것을 추천합니다.

3. **(우선순위: 하) 보정 로그(`corrections.log`) 시각화 개선**
   - 단순 줄 대조 방식 외에도 `difflib` 라이브러리를 활용해 변경된 어절을 강조하거나 디프(diff) 형태로 기록하도록 구현하면 사용자가 보정본을 검수하기 훨씬 용이해집니다.
