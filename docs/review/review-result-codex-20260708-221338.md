### 종합 평가
- 전체 점수: 7/10
- 한줄 요약: 큰 방향은 좋지만, API 계약·작업 상태 복구·CLI 보정 실패 처리·보안 경계가 아직 구현 직전 설계로는 부족합니다.

| 관점 | 평가 | 상세 |

### 구체적 피드백
1. [Blocker] API 응답 구조가 너무 추상적입니다.  
   “프런트 계약 그대로”라고 되어 있지만, 실제 `Job`, `PageFile`, `JobSummary`, `PageDetail` 필드 목록이 문서에 없습니다. 구현자가 프런트 타입을 다시 열어봐야 해서 설계 문서만으로 구현이 불가능합니다.

2. [Blocker] 서버 재시작 후 상태 정책이 애매합니다.  
   “메모리는 소멸, 디스크 산출물은 남아 다운로드 가능”이라고 했지만, `GET /api/jobs/:id`가 메모리에 없으면 404라고 되어 있습니다. 그러면 남은 산출물을 실제로 다운로드할 방법이 없습니다.

3. [Blocker] `retry`가 책 전체 조립·보정과 충돌합니다.  
   `POST /api/jobs/:id/retry/:fileId`가 실패 페이지를 다시 OCR한다고 되어 있지만, 이후 `book.txt`, `book_corrected.txt`, `corrections.log`를 다시 만들지 명확하지 않습니다.

4. CLI 보정 백엔드의 실패 기준이 부족합니다.  
   `codex`나 `claude`가 장황한 설명, 빈 응답, 부분 응답, 마커 순서 변경, 중복 문단을 반환할 수 있습니다. 개수 불일치만으로는 방어가 약합니다.

5. 업로드 검증이 부족합니다.  
   jpg/jpeg만 받는다고 했지만 MIME 타입, 확장자, 파일 크기, 총 페이지 수, 손상 이미지, 중복 파일명 처리 기준이 없습니다.

6. 동시 작업 제한이 없습니다.  
   단일 사용자라도 여러 잡을 동시에 올리면 OCR, CLI, 디스크 I/O가 겹칩니다. `ThreadPoolExecutor`의 최대 작업 수와 큐 제한이 필요합니다.

7. 보정 옵션의 모델 의미가 불명확합니다.  
   `backend=codex`는 `gpt-5.5`, `claude=기본 모델`이라고 되어 있는데, `model` 파라미터를 받는 API와 충돌합니다. 사용자가 모델을 직접 고를 수 있는지 정해야 합니다.

8. 보안 경계가 약합니다.  
   개인용이어도 파일 업로드 서비스입니다. 파일명 경로 조작, 대용량 업로드, 임시 파일 정리, CORS 허용 범위는 최소한 명시해야 합니다.

9. 테스트 범위가 문서 지침보다 넓습니다.  
   `JobStore`, 단순 API 계약 테스트까지 많이 넣으면 과합니다. 핵심은 배치 보정 파싱, 파이프라인 실패 보존, 재시도 후 재조립입니다.

10. “보정 실패 시 원문 유지”는 좋지만 사용자 표시가 부족합니다.  
   어느 문단이 실패했는지, 전체 보정 중 몇 개가 원문 유지됐는지 결과 화면이나 로그에서 보여줘야 검수할 수 있습니다.

### 개선 제안
1. 우선 API 스키마를 문서에 고정하세요.  
   `Job.status`, `Job.phase`, `files[].status`, `summary`, `correction`의 실제 JSON 예시를 넣는 것이 가장 먼저입니다.

2. 재시작 정책을 둘 중 하나로 정하세요.  
   추천은 “최소 구현에서는 재시작 후 과거 잡 조회 불가, 다운로드도 불가”입니다. 단순하고 일관됩니다. 다운로드 복구까지 하려면 메타데이터 JSON 저장이 필요합니다.

3. `retry`는 범위를 줄이세요.  
   추천은 v1에서 “실패 페이지 재시도 후 book.txt까지만 재생성, 보정본은 다시 만들지 않고 보정 재실행 버튼은 추후”입니다.

4. CLI 출력은 JSON Lines 형식으로 강제하세요.  
   마커 텍스트 파싱보다 `{ "id": 1, "text": "..." }` 배열을 요구하고 JSON 파싱 실패 시 단건 재시도하는 편이 안전합니다.

5. 업로드 제한을 명시하세요.  
   예: jpg/jpeg만 허용, 파일당 20MB, 한 잡 최대 100장, 전체 500MB, 파일명은 저장 시 내부 안전 이름으로 변경.

6. 작업 제한을 넣으세요.  
   예: 동시 잡 1개, 보정 CLI 동시성 기본 2개, 큐 최대 3개. 초과 시 429 또는 400으로 거절합니다.

7. 테스트는 핵심 3개만 강하게 가져가세요.  
   배치 보정 파싱 실패 폴백, OCR 일부 실패 후 book.txt 보존, 보정 백엔드 미가용 시 잡 done + warning 이 세 가지가 제일 중요합니다.
