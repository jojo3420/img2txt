### 구체적 피드백
1. [Blocker] 꼬리말 제거 규칙이 위험합니다.  
   “최하단 띠 안의 줄을 모두 제거”하면 본문 마지막 줄이 낮게 찍힌 경우 본문이 삭제될 수 있습니다. 최소한 `위치 + 짧은 줄 + 페이지 번호/Chapter/책 제목 패턴` 같은 보조 조건이 필요합니다.

2. [Blocker] “꼬리말을 못 찾으면 제거하지 않음”과 “최하단 띠 안 줄 모두 제거”가 충돌합니다.  
   지금 규칙만 보면 최하단 띠에 줄이 있으면 무조건 꼬리말로 볼 수 있습니다. “꼬리말 후보 판정 기준”을 따로 정의해야 합니다.

3. `correct` 종료 코드 기준이 애매합니다.  
   시작 전 Ollama 연결 실패는 `1`, 실행 중 문단 실패는 `0`은 타당합니다. 하지만 “모델 없음”, “중간에 Ollama 죽음”, “전체 문단 실패”를 각각 어떻게 볼지 정해야 합니다.

4. LLM 보정 로그 포맷이 부족합니다.  
   `corrections.log`에 문단 번호, 모델명, 변경 전/후, 가드 차단 여부, 실패 이유가 있어야 검수가 쉽습니다.

5. 긴 문단 처리 전략이 없습니다.  
   로컬 LLM도 입력 길이 제한이 있습니다. 너무 긴 문단은 나누거나 원문 유지하는 기준이 필요합니다.

6. 출력 파일 덮어쓰기 정책이 없습니다.  
   기존 `output/book.txt`가 있을 때 덮어쓸지, 실패할지, `--force`를 둘지 정해야 합니다.

7. 인코딩 기준이 빠져 있습니다.  
   모든 txt/log는 `UTF-8`로 고정하는 것이 좋습니다.

8. 파일 확장자 범위가 좁습니다.  
   문서에는 `jpg`만 있습니다. 실제로는 `.jpg`, `.jpeg`, 대소문자 확장자까지 처리할지 정해야 합니다.

9. 자연 정렬 기준이 더 필요합니다.  
   “파일명 마지막 숫자”는 좋지만, 숫자가 여러 개 있는 파일명 예외를 문서에 예시로 박아두는 편이 안전합니다.

10. 테스트 전략이 약간 과합니다.  
   사용자 원칙대로라면 단순 scanner/writer류 테스트는 줄이고, 핵심인 `layout`, `assembler`, `corrector`에 집중하는 편이 낫습니다.

1. 1순위: 꼬리말 판정 규칙을 명확히 분리하세요.  
   추천: `bottom band 안에 있음` + `짧은 줄` + `페이지 번호 또는 Chapter/책 제목 유사 패턴`을 만족할 때만 제거.

2. 2순위: `correct` 실패 기준을 정리하세요.  
   추천: 시작 전 연결/모델 확인 실패는 `1`, 실행 중 일부 실패는 `0`, 전체 문단 실패는 `1`로 두는 것이 명확합니다.

3. 3순위: 로그 포맷을 고정하세요.  
   추천: `paragraph_id`, `status`, `model`, `before`, `after`, `reason` 구조로 남기면 나중에 검수하기 쉽습니다.

4. 4순위: 긴 문단과 출력 덮어쓰기 정책을 추가하세요.  
   추천: 긴 문단은 원문 유지 + warning, 기존 출력은 기본 덮어쓰기 금지 + `--force` 허용.

5. 5순위: 테스트 범위를 줄이세요.  
   핵심 테스트는 `layout`, `assembler`, `corrector`에 집중하고, 단순 파일 쓰기/읽기는 최소 검증만 두는 편이 실용적입니다.
