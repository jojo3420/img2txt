### 동의하는 핵심 피드백 (Top 3)
1. 상태별 다운로드 응답 계약(지적 2): 보정 미실행/실패 시 corrected-log 다운로드 요청 결과가 스펙에 없음. 화면 연결에서 바로 막히는 실질 공백.
2. Vite dev proxy 분리(지적 7): 단일 명령 실행과 프런트 개발 반복은 별개 요구. proxy를 dev용으로 되살리면 둘 다 만족.
3. retry 404/409 분리(지적 8): 코드로 확인한 실제 문제. 라우트 응답 정확도에 직접 영향.

### 동의하지 않는 피드백
- 지적 5(backend 미지원 가능성): factory가 claude/codex 실제 지원 확인. 노출 유지.
- 지적 10(원본명 미보존 우려): create_job이 원본명 보존 확인. 구현 가능.

### Codex가 놓친 추가 관점 (Claude 보완)
1. 대용량 업로드와 이벤트 루프 블로킹: `POST /api/jobs`를 async로 두고 내부에서 500MB 저장(create_job)까지 동기 수행하면 이벤트 루프가 막힌다. 업로드 라우트는 동기 `def`로 선언해 FastAPI가 스레드풀에서 처리하게 하거나, 파일 저장을 오프로딩해야 한다.
2. Content-Disposition 하위호환: `filename*`(RFC 5987)만 두지 말고 ASCII `filename=` 폴백을 함께 제공해 구형 클라이언트 대비.

## 3. 최종 권고사항

### 즉시 반영 (Must, P0)
1. 다운로드 상태 계약 표 추가: 산출물 파일 부재(보정 스킵/실패/생성 전)→404, 정상→200 스트림, correctedStale→200(Job 필드로 표식). 화면 표시는 fetch로 /download 본문 읽기 명시.
2. 옵션 계약 고정: backend는 codex/claude(claude 실측 미검증 표기), model은 백엔드 종속 읽기전용, correct=false면 backend/model 무시. 공통 오류 JSON 형태(`{detail: ...}`) 정의.
3. 업로드 라우트를 동기 `def`로(이벤트 루프 블로킹 회피) + 용량초과 413 추가, 나머지 위반 400 유지.

### 검토 후 반영 (Should, P1)
1. Vite dev proxy 되살리기(dev만), run/prod는 same-origin 유지.
2. retry: get_job 404 선검사 → retry_file False→409.
3. JPEG 매직바이트+빈 파일 검증. SPA 폴백은 화면 경로만(자산 미스 404).
4. JobStore 접근은 app.state 주입으로 명시(테스트에서 임시 JOBS_ROOT 주입 가능하게).
5. page n=pageNumber 명시. 프런트 phase/correctionError/correctedStale 표시 규칙 보강.

### 참고 (Nice to have, P2)
1. 의존성 위치(pyproject)+공식 실행 절차(build→uvicorn) 문서화.
2. Content-Disposition ASCII 폴백 병기.
3. 동시 업로드 메모리는 단일 사용자 전제 하 알려진 한계로 기재.
