### 구체적 피드백

1. **[Blocker] CLI 프로세스 타임아웃 및 좀비 프로세스 방지 책 미비**
   - `claude` 및 `codex` CLI를 subprocess로 호출할 때, 네트워크 지연이나 세션 만료로 인해 프로세스가 멈추는(Hang) 현상이 발생할 수 있습니다. 
   - 서브프로세스 호출 시 반드시 명시적인 `timeout`을 지정해야 하며, 예외 발생 시 생성된 하위 프로세스를 강제 종료(`terminate`/`kill`)하여 좀비 프로세스가 쌓이는 것을 방지하는 안전장치가 아키텍처 설계에 포함되어야 합니다.

2. **[Blocker] 업로드 제한 및 디스크 공간 누수 예외 처리 누락**
   - 책 스캔 이미지(수십~수백 장)를 한 번에 업로드하므로 메모리 과부하(FastAPI `UploadFile` 처리 방식 조율 필요) 및 용량 초과 문제가 발생할 수 있습니다. 단일 요청 업로드 파일 개수 및 총 용량 제한(Max Payload Size) 정의가 필요합니다.
   - 잡 메타데이터는 메모리(`JobStore`)로 관리하여 서버 재시작 시 사라지지만, 업로드된 원본 이미지와 산출물은 디스크에 계속 유지됩니다. 자동 정리 정책이 없으면 디스크 공간이 빠르게 고갈되므로, 서버 기동 시 또는 주기적으로 하루 이상 지난 `./.data/jobs/<job_id>` 디렉터리를 비우는 정리 태스크가 최소한으로 필요합니다.

3. **[Blocker] CLI 실행 환경의 환경 변수(Auth Context) 유실 리스크**
   - `claude` 및 `codex` CLI는 사용자의 터미널 쉘 세션의 인증 정보(예: `~/.config`, 환경 변수, OAuth 토큰)를 사용합니다.
   - FastAPI가 백그라운드로 실행될 때 시스템 데몬 형태로 돌거나 독립된 환경에서 실행되면, 상위 쉘의 환경 변수(`PATH`, `HOME` 등)가 인계되지 않아 CLI 인증 오류가 발생할 가능성이 매우 높습니다. 백엔드에서 서브프로세스를 기동할 때 호출 프로세스의 환경 변수를 그대로 전파(`env=os.environ`)하도록 명시해야 합니다.

4. **[주의] 특정 페이지 재-OCR(`POST /api/jobs/:id/retry/:fileId`) 흐름의 모호성**
   - 전체 변환 파이프라인은 `이미지 업로드 -> OCR -> 조립 -> (옵션) 보정` 순서로 일괄 실행됩니다.
   - 특정 페이지만 재-OCR 했을 때, 이미 완성된 `book.txt` 및 `book_corrected.txt`에 해당 페이지의 텍스트를 어떻게 다시 끼워 넣을 것인지, 그리고 보정 단계도 해당 페이지 영역에 대해 다시 실행할 것인지에 대한 재조립 파이프라인 설계가 구체화되어야 합니다.

5. **[주의] 프런트엔드 UI/UX 복잡도 증가**
   - 페이지별 보정본(`PageDetail.corrected`)을 비우고 책 전체 단위(`book_corrected.txt`)로만 원본/보정본 비교 뷰를 제공하도록 프런트를 조정하겠다고 했습니다.
   - 책 전체 텍스트는 분량이 매우 클 수 있으므로, 웹 브라우저에서 한 번에 렌더링하고 비교(Diff)할 때 메모리 저하 및 렉이 발생할 수 있습니다. 텍스트가 클 경우 화면 성능에 대응할 UI 렌더링 전략(예: 가상 스크롤, 100줄 단위 페이징 등)이 고려되어야 합니다.

### 개선 제안

1. **[우선순위: 상] 정식 API 백엔드(API Key 방식) 폴백 구현 및 주입**
   - CLI 백엔드는 개인용으로 유용하지만 인증 환경이 변경되거나 레이트 리밋에 걸리기 쉽습니다. API Key(예: `ANTHROPIC_API_KEY`)가 환경 변수로 감지될 경우 CLI 대신 상용 SDK나 direct HTTP 호출을 통해 OpenAI/Anthropic API를 사용하는 `API-based Backend`를 추상화 인터페이스 하위에 나란히 설계해두면 ToS 리스크 및 CLI 환경 리스크를 즉시 회피할 수 있습니다.

2. **[우선순위: 중] 문단 마커 매칭 실패 시 단건 폴백 범위 구체화**
   - 배치 처리 시 마커 개수가 맞지 않아 단건으로 쪼개서 재시도할 때, 무한 루프나 과도한 API 호출로 비용 및 시간이 늘어나는 것을 방지하기 위한 최대 단건 재시도 횟수 제한(Max Retry Guard)을 두는 것을 추천합니다.

3. **[우선순위: 하] 다운로드 파일 아카이빙**
   - `book.txt`, `book_corrected.txt`, `corrections.log` 외에도 개별 이미지 및 페이지별 OCR 결과 등을 포함하여 한 번에 다운로드받을 수 있는 `GET /api/jobs/:id/download?type=zip` 기능이 추가된다면 사용성이 크게 향상될 것입니다.
