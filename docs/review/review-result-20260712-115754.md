### 검증 결과
- 판정: 미흡
- 한줄 요약: P1은 대부분 반영됐지만, 다운로드 상태·모델 조합·공통 오류 계약에 P0가 남아 있다.

### P0/P1 반영 점검
| 항목 | 반영 여부 | 근거/미흡점 |
|------|-----------|-------------|
| P0-1 다운로드 상태별 응답 | 부분 반영 | 파일 존재 여부만 본다. 보정 생성 중·미실행·실패를 정확히 구분하지 못한다. |
| P0-2 backend/model/correct 계약 | 부분 반영 | backend는 고정했지만, `model`을 폼으로 받는지 서버가 만드는지 모순이다. Claude 모델값도 미정이다. |
| P0-2 공통 오류 JSON | 미흡 | FastAPI 입력 검증 오류는 기본적으로 422와 `detail` 배열이다. 문서의 400·문자열 형식과 다르다. 500 JSON 처리도 정의되지 않았다. |
| P0-3 이벤트 루프 블로킹 | 반영 | 업로드 라우트를 동기 `def`로 선언해 파일 처리 전체를 스레드풀에서 실행한다. |
| P1 Vite 개발 proxy | 반영 | 개발 서버의 `/api` proxy가 명시됐다. |
| P1 retry 404/409 분리 | 부분 반영 | 잡 없음은 404지만, 존재하지 않는 `fileId`도 409가 된다. “미존재 페이지는 404” 규칙과 맞지 않는다. |
| P1 JPEG 매직바이트 | 반영 | `FF D8 FF` 검증이 명시됐다. |
| P1 SPA 자산 404 | 반영 | 확장자 있는 미존재 자산은 404로 처리한다. |
| P1 JobStore 주입 | 반영 | `app.state.job_store`와 의존성 주입 방식이 명시됐다. |
| P1 `n=pageNumber` | 반영 | 배열 위치가 아닌 `PageFile.pageNumber`라고 명시됐다. |
| factory 실제 지원 정정 | 반영 | codex·claude 지원 사실이 반영됐다. |
| 원본 파일명 보존 정정 | 반영 | `PageFile.filename`을 다운로드 이름의 근거로 사용한다. |

### 남은 블로커 (있으면 P0로, 없으면 "없음")
1. `model` 요청 계약을 하나로 확정해야 한다. 폼 필드에서 제거하고 서버에서 파생할지, 사용자가 보내도록 할지 선택해야 한다. 서버 파생이라면 codex·claude별 실제 저장값을 명시해야 한다.
2. 보정본 다운로드에서 생성 중·보정 미실행·보정 실패를 `Job.phase`, `correct`, `correctionError`로 구분하고 각각의 응답을 정의해야 한다.
3. 입력 검증 오류와 서버 오류의 상태 코드·JSON 형식을 확정해야 한다. 특히 FastAPI 기본 422를 유지할지, 예외 처리기로 400 `{"detail":"..."}`로 바꿀지 명시해야 한다.

### 참고(P2)
- 업로드 파일은 한 번에 전부 읽지 말고 제한 크기 단위로 읽어야 용량 제한을 실제로 지킬 수 있다.
- 파일명 정제 시 빈 이름, 지나치게 긴 이름, 제어문자 처리도 정하면 안전하다.
- 프런트가 `fetch`로 다운로드한다면 `Content-Disposition` 파일명을 직접 파싱해야 한다.
