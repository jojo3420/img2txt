## High 이슈 — Should-fix

### H1. 재시도 성공 후 `book_corrected.txt`·`corrections.log`·보정 통계가 낡은 채 방치됨 (Silent Failure)

- **파일:라인:** `server/pipeline.py:360-439` (`retry_page_pipeline`), 대조 `server/pipeline.py:330-349`
- **문제:** `retry_page_pipeline`은 `pages/page-NNN.txt`, `layouts/page-NNN.json`, `book.txt`만 교체하고 `book_corrected.txt`·`corrections.log`·`job.summary.corrected/kept/guardBlocked`는 전혀 손대지 않습니다.
- **재현 흐름:**
  1. `correct=True`로 이미지 2장 업로드, 2페이지 OCR 실패 → `book.txt`에 `[페이지 2 누락]`, 보정 실행 → `book_corrected.txt` 생성, 잡 `done`.
  2. 사용자가 2페이지 재시도 성공 → `book.txt`는 "복구 2" 반영되지만 `book_corrected.txt`는 여전히 `[페이지 2 누락]`.
  3. 잡 상태는 `done`, `correctionError`는 `None`. 아무 경고 없이 **복구된 페이지가 빠진 보정본**을 사용자가 내려받게 됨.
- **근거:** `book_corrected.txt`를 쓰는 코드는 `run_correct_pipeline`(330행)뿐이고 재시도 경로에는 없음. `job.summary.corrected` 등도 316-349행에서만 설정됨. 스펙 "범위 밖: 보정본 자동 재생성"으로 의도된 한계이나, **낡음을 알리는 신호가 전혀 없다는 것**이 문제입니다.
- **제안 수정(둘 중 하나):**
  - 최소안: 재시도 성공 시 `job`에 보정본 무효화 신호를 남긴다. 예) `job.correctionError = "재시도 이후 보정본이 최신이 아닙니다. 재보정이 필요합니다"` 또는 `book_corrected.txt`/`corrections.log`를 삭제해 다운로드 계약에서 "보정본 없음"으로 처리.
  - 근본안: `correct` 옵션 잡의 재시도 성공 후 변경된 페이지만 재보정하거나 전체 재보정을 트리거. (범위가 크면 T10 다운로드 계약에서 stale 플래그를 노출)

## 긍정적 관찰

- **트랜잭션 설계가 견고합니다.** 임시파일 → `fsync` → `os.replace` → 디렉터리 `fsync`에 마커 기반 롤백까지 더해, 커밋 중 fsync 실패(테스트 2)와 프로세스 중단(테스트 3, `BaseException`으로 kill 모사)을 모두 복구합니다. 테스트가 실패 주입으로 이를 실제 재현하는 점이 특히 좋습니다.

- **경로 보안이 탄탄합니다.** 디스크 경로는 서버 생성 내부명(`upload-<id>-page-NNNN.jpg`)만 쓰고 사용자 원본명은 메타데이터로만 보관, `sanitize_filename`이 이중 방어합니다. 중복 원본명 덮어쓰기와 경로 조작을 함께 차단합니다.
