### 종합 평가
- 전체 점수: **7.5/10**
- 한줄 요약: **큰 방향은 타당하지만, “델타면 읽기순서 휴리스틱 오류가 상쇄된다”는 전제가 약하고, 지표별 판정 규칙이 아직 충분히 단단하지 않습니다.**

### 구체적 피드백
1. **두 지표 병행 설계는 방향이 맞습니다.**  
   놓침률은 “OCR이 정답 글자를 얼마나 빠뜨렸나”를 보는 품질 하한 게이트로 적합합니다. 읽기순서 CER 델타는 LLM 보정이나 전처리가 글자 순서를 더 망쳤는지 보는 보조 신호로 쓸 수 있습니다.

2. **[Blocker] 읽기순서 CER 델타를 “랭킹 지표”로 쓰는 근거가 약합니다.**  
   델타는 같은 페이지에서 `assembled → corrected` 변화를 보는 데는 좋습니다. 하지만 서로 다른 모델 비교에서는 모델마다 출력 순서, 추가 텍스트, 보정 방식이 달라져 상수 편향이 완전히 상쇄된다고 보기 어렵습니다.  
   즉 “상쇄된다”가 아니라 “같은 입력/같은 참조 기준의 paired 비교에서는 일부 완화된다” 정도로 낮춰 써야 합니다.

3. **[Blocker] 다열/표 양식에서 y-밴드 그룹핑은 오작동 가능성이 큽니다.**  
   예를 들어 좌우 2열 문서, 표 안의 여러 셀, 체크박스+라벨, 제목/본문 높이가 다른 문서에서는 `중앙값 높이 × 0.6` 기준이 행을 잘못 묶을 수 있습니다.  
   이 실패가 델타로 항상 상쇄된다는 논리는 성립하지 않습니다. OCR/LLM 출력이 실제 읽기순서와 다르면 델타 자체도 흔들립니다.

4. **놓침률 산식은 품질 하한으로는 타당합니다.**  
   공백 제외 글자 multiset 방식은 읽기순서 착시를 제거합니다. `분모=정답 글자수`도 직관적입니다.  
   다만 위치 오류, 필드 이동, 의미 오류, 같은 글자 반복 오류는 못 잡습니다. 문서에도 적혀 있지만, 이 한계를 더 강하게 명시해야 합니다.

5. **추가율 산식도 진단용으로는 타당합니다.**  
   OCR/LLM이 정답에 없는 글자를 얼마나 많이 만들었는지 보는 데 좋습니다.  
   다만 라벨이 불완전하면 “정확히 읽은 양식 헤더”도 추가로 잡힐 수 있습니다. 그래서 합격/불합격 기준으로 쓰면 위험합니다. 현재처럼 진단용으로 제한하는 게 맞습니다.

6. **`N=0이면 0.0`은 위험합니다.**  
   빈 정답인데 OCR이 많은 글자를 냈다면 추가율이 0으로 나와버립니다. 이건 환각을 숨깁니다.  
   최소한 `empty_ref_extra_count`, `empty_ref_has_output` 같은 별도 필드가 필요합니다. 아니면 `N=0 and hyp>0`일 때 추가율을 별도 sentinel 값으로 다뤄야 합니다.

7. **정규화 기준이 부족합니다.**  
   NFC와 공백 정리만으로 충분한지 불명확합니다. OCR에서는 전각/반각, 괄호, 하이픈, 원문자, 한자/숫자 혼동이 자주 나옵니다.  
   엄격 평가라면 지금 방식이 맞지만, “왜 strict만 쓰는지”를 문서에 명확히 적어야 합니다.

8. **micro 집계만 있으면 긴 페이지가 결과를 지배합니다.**  
   micro는 전체 글자 기준이라 합리적입니다. 하지만 페이지별 품질 저하를 놓칠 수 있습니다.  
   summary에 macro 평균이나 worst page 목록까지는 아니더라도 `page_count`, `p50`, `p90` 정도를 검토할 만합니다.

9. **LLM 모델 비교 소비자에게는 현재 지표만으로 부족합니다.**  
   LLM 보정은 글자를 맞히는 것 외에 “없는 정보 생성”, “필드 간 값 이동”, “형식 보존 실패”가 중요합니다.  
   놓침률+추가율+CER 델타는 기본 체온계로는 좋지만, 최종 모델 랭킹 기준으로는 약합니다.

10. **전처리 재평가 소비자에게는 꽤 타당합니다.**  
   전처리는 보통 OCR 입력 품질을 바꾸는 작업이라 놓침률 하락 여부가 핵심입니다.  
   다만 전처리가 노이즈까지 잘 읽게 만들어 추가율을 올릴 수 있으므로, `miss_rate 개선 + extra_rate 악화 없음` 형태의 판정이 필요합니다.

11. **테스트 계획은 좋지만 실패 케이스가 더 필요합니다.**  
   현재 테스트는 정상 행 정렬 중심입니다. 다열, 표, 높이가 다른 제목/본문, x/y 길이 불일치, 음수/0 height, NaN 좌표 같은 케이스를 추가해야 합니다.

12. **문서의 역할 표현을 정리해야 합니다.**  
   추천 역할은 이렇습니다.  
   놓침률: 품질 하한.  
   추가율: 환각/라벨 불완전성 진단.  
   읽기순서 CER 델타: 순서/문장성 부작용 감시.  
   모델 최종 랭킹: 비용, 속도, 놓침률, 추가율, CER 델타를 묶은 별도 판정표.

### 개선 제안
1. **최우선: 델타 상쇄 논리를 약화하고 조건을 명시하세요.**  
   “상수 편향이 상쇄된다”가 아니라 “같은 페이지, 같은 참조, 같은 입력 계열의 paired 비교에서는 편향 영향이 줄어든다”로 바꾸는 게 안전합니다.

2. **최우선: 빈 정답 처리 정책을 바꾸세요.**  
   `N=0이면 extra_rate=0`은 위험합니다. 빈 정답에서 출력이 있으면 별도 오류로 기록해야 합니다.

3. **높은 우선순위: 읽기순서 재정렬 실패 감지 필드를 추가하세요.**  
   예: `reading_order_method`, `row_count`, `median_height`, `bbox_count`, `suspicious_layout_flag`.  
   나중에 이상치 페이지를 추적할 수 있습니다.

4. **높은 우선순위: 전처리 판정 규칙을 명시하세요.**  
   예: “놓침률이 개선되고, 추가율과 degraded_page_count가 악화되지 않으면 채택 후보”처럼 써야 합니다.

5. **중간 우선순위: LLM 모델 비교는 단일 랭킹보다 게이트+비교표로 가세요.**  
   먼저 놓침률로 탈락선을 만들고, 그다음 추가율/CER 델타/비용/속도를 함께 보는 방식이 더 안전합니다.

6. **중간 우선순위: 다열/표 실패 케이스를 문서에 명시하세요.**  
   지금은 “YAGNI”라고 되어 있지만, 이 데이터가 2차원 양식이라면 바로 핵심 리스크입니다. 고도화는 안 하더라도 실패 가능성은 테스트와 리포트에 남겨야 합니다.

7. **낮은 우선순위: macro 통계나 분위수 지표를 추가 검토하세요.**  
   micro만 있으면 특정 긴 문서가 전체 결과를 끌고 갈 수 있습니다. p50/p90 정도만 있어도 해석력이 좋아집니다.
